Architecture Decision Summary
Pros of Client-Side Rendering:
  • ✓ No full page reload for sorting/filtering/pagination
  • ✓ Smaller JSON payloads vs full HTML (typically 60-70% smaller)
  • ✓ Better UX for interactive features (calculated fields, live validation)
  • ✓ Enables client-side sorting/filtering without round-trips
Cons of Client-Side Rendering:
  • ✗ Requires JavaScript (breaks without it unless progressive enhancement)
  • ✗ Adds ~6-9 KB to initial JS bundle
  • ✗ Slower first paint (JS must parse/execute before content)
  • ✗ More complexity in codebase (two rendering paths)
  • ✗ Conflicts with "bare metal" simplicity philosophy
Recommendation:

DEFER for now. BareMetalWeb's philosophy prioritizes simplicity, understanding, and server-side speed. The current server-side rendering is already brutally fast (0.1-0.15ms). Client-side rendering would add complexity and conflict with the zero-magic ethos.

Reconsider if: Interactive features like calculated fields (#66), bulk operations (#64), or live lookups (#58) become primary use cases and users demand richer interactivity beyond what progressive enhancement can provide.
